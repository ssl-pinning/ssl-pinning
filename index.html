ssl-pinning / server

Dynamic SSL Pinning
Server

A service that continuously tracks TLS certificate public keys of target HTTPS endpoints, stores fingerprints in a configurable backend, and exposes them as a cryptographically signed JSON registry — enabling pin rotation without app updates.

Docker Go 1.26+ Prometheus BSD-3-Clause

Getting started

1

Generate RSA signing keys

The server signs the pin registry with a 4096-bit RSA private key. The public key is embedded in client SDKs for signature verification.

openssl genpkey -algorithm RSA -pkeyopt rsa_keygen_bits:4096 -out prv.pem
openssl rsa -pubout -in prv.pem -out pub.pem
Important: keep prv.pem secret — it never leaves the server. Only pub.pem is distributed to clients.
2

Create config.yaml

keys:
  - fqdn: api.example.com

server:
  listen: 0.0.0.0:7500

storage:
  type: memory

tls:
  dir: /etc/ssl-pinning/tls
  dump_interval: 30s
  timeout: 10s
3

Run with Docker

docker run -d \
  -v $(pwd)/config.yaml:/etc/ssl-pinning/config.yaml \
  -v $(pwd)/prv.pem:/etc/ssl-pinning/tls/prv.pem \
  -p 7500:7500 -p 9090:9090 \
  ghcr.io/ssl-pinning/ssl-pinning:latest up
4

Verify

curl http://localhost:7500/api/v1/api.example.com.json
Kubernetes: use the Helm chart for production deployments.

How it works

Traditional SSL pinning relies on static fingerprints embedded in clients. When certificates rotate, pins go stale. ssl-pinning solves this with a dynamic registry.

1

Server periodically dials each configured FQDN on port 443 and extracts the public key from the TLS certificate chain.

2

Fingerprints are computed and stored in the configured storage backend (memory, filesystem, Redis, or Postgres).

3

The JSON payload is cryptographically signed with the RSA private key before being served to clients.

4

Client SDKs fetch the registry, verify the RSA signature, and enforce pinning — even across certificate rotations.

5

Even if an attacker obtains a fraudulent certificate for a target domain, they cannot forge a valid signed fingerprint list.

Keys configuration

Each entry under keys defines a domain to track. Multiple entries can share the same file to group them into one API response.

FieldDefaultDescription
fqdn—Required. Domain to dial on port 443 for certificate tracking.
domainName*.{fqdn}Hostname pattern for certificate pinner on the client side.
file{fqdn}.jsonGroups this entry into a named response file. Multiple FQDNs can share the same file.
 
keys:
  - fqdn: google.com
    domainName: "www.google.com"
  - fqdn: mail.google.com
    file: google.com.json      # grouped into the same endpoint
    domainName: "mail.google.com"

Both entries are served at GET /api/v1/google.com.json

Configuration reference

All settings can be set via config file, environment variable, or CLI flag. Priority: flags > env vars > config file > defaults. Env var format: SSL_PINNING_ prefix + upper snake case (e.g. log.level → SSL_PINNING_LOG_LEVEL).
